The artefact will be catered towards the requirements in the brief in several ways

How will the website meet the brief

Requirements to be met:

  • 1 or more data capture forms
  • Implement validation checks
  • Accessibility
  • GDPR and acceptable use
    • Designed to comply with GDPR, user passwords are hashed and there is nothing within my artefact which will identify me, the windows account I used was not on my own name and the github repo I used for some images was connected with a throwaway account.
1 or more data capture forms:

The website will contain 3 data capture forms, 1 newsletter, 1 login page and 1 sign up page. All of their data will be captured via Flask. The site will programmed so that anytime a POST request is made any data sent in that request is retrieved.

Implement validation checks:

I plan to use python functions to implement validation checks. This should be easy as flask will allow me to use python with my website quite easily. I can implement the checks in the same area where the data is retrieved. For example if a certain value needs to be greater than 5 characters I can create an if statement to validate it. If all a user's info is valid I can use flask to render the template the user is requesting or I can use the 'login_user()' flask command. Bootstrap will also allow me to implement validation checks by adding a 'type' to certain inputs, if the type is set to email bootstrap will automatically notify the user that '@' is required for the email field.

Accessibility:

Throughout the site I plan to use bootstrap functions that make the website responsive to all different screen sizes, so no matter what platform/browser a user is on the website will still be functional. In this image we see an example of a navbar that only expands out when viewed on a larger screen, otherwise it is contracted inside of a navbar toggler. It is also possible in bootstrap to make certain images dissapear if the webpage is viewed on a smaller screen. This is done using (d-sm-block)

How the embedded system met the brief:

Requirements to be met:

  • Must be a standalone device (remote operation is not needed)
  • Operates with little to no manual intervention
  • Can be monitored on the website in some way

System architecture (face recognition and identification)

The flowchart on the right describe the system architecture of the face recognition and identification process. Once the code is run, the process will continously loop (therefore is is fully automated as per the brief) until the user manually interrupts it by pressing the 'q' key. While the code is running, the laptop displays a colour version of the camera feed, and converts a seperate frame into grayscale, as opencv's cascade recognizers and face identification functions require the image to be in grayscale. Using the frontal_face cascade, the program checks if it is able to identify a frontal face anywhere in the frame, and if it can it will generate the co-ordinates of that face, and display them on the colour frame which was initialized in the beginning. The program will then further attempt to identify a face in the frame, by comparing it to the images provided by the user in "User data", it can then retrieve the name of that face and also display that on the colour frame. This means the embedded system will be a standalone system and will require no manual intervention once it is running.

Data flow

As seen in the data flow diagram on the right, when face recognition and identification occurs the data developed is sent and stored in firebase. This way the embedded system can be monitored from the website as the brief says.
